## P0 - Critical（必須在開放測試前修復）

### 1. [安全] Slack OAuth 登入不檢查帳號是否停用
- **檔案**: `backend/src/services/authService.ts:76-153`
- **問題**: `loginWithSlackCode()` 找到帳號後直接核發 JWT，未檢查 `isActive`。已停用的帳號可透過 Slack OAuth 重新登入。`devLogin()` 有正確檢查，但 Slack OAuth 遺漏了。
- **影響**: 離職員工可繞過停用機制，取得系統完整存取權
- **修復**: 在找到 employee 後加入 `if (!employee.isActive) throw new AppError(403, "Account is disabled")`

### 2. [安全] Demo 登入按鈕在所有環境永久可見
- **檔案**: `packages/frontend/src/pages/LoginPage.vue:124`
- **問題**: 「Demo 模式快速登入」按鈕無條件顯示，配合 `ENABLE_DEV_LOGIN=true` 任何訪客都能以固定帳號 `emp-6` 一鍵登入
- **影響**: 測試環境中任何知道網址的人都能未經授權登入
- **修復**: 加環境判斷 `v-if="isDemoEnvironment"` 或使用環境變數控制顯示

### 3. [後端] Slack 簽名驗證使用 JSON.stringify 重建 body，驗證永遠失敗
- **檔案**: `backend/src/routes/slack.ts:38`
- **問題**: `verifySlackSignature` 用 `JSON.stringify(req.body)` 計算 HMAC，但 JSON.stringify 的格式與 Slack 原始 body 不同，等於簽名驗證形同虛設。程式碼自己也承認問題但未修復。
- **影響**: 任何人可偽造 Slack 指令查詢員工工時
- **修復**: 用 `express.raw()` 保留原始 body 或改用 `@slack/bolt` SDK

### 4. [前端] TaskRelationCard 硬寫死 Mock 資料
- **檔案**: `packages/frontend/src/components/task/TaskRelationCard.vue:7,27`
- **問題**: 直接 `import { mockUsers } from '@/mocks/employees'`，在 API 模式下永遠顯示錯誤的負責人資訊
- **影響**: 內測時任務關聯卡片的負責人欄位全部空白或顯示錯誤
- **修復**: 改用 `useEmployeeStore().getEmployeeById()` 取得真實資料

### 5. [資料庫] GitLab convertToTimeEntry 的 create + update 不在同一個 transaction
- **檔案**: `backend/src/services/gitlab/activityService.ts:369-389`
- **問題**: 先 `prisma.timeEntry.create()` 再獨立 `prisma.gitLabActivity.update()`，第二步失敗會產生孤兒 time entry，重試時可能重複建立
- **影響**: 資料不一致，工時記錄可能重複
- **修復**: 用 `prisma.$transaction()` 包住整個操作

## P1 - High（應在開放測試前或測試初期修復）

### 6. [後端] JWT TokenExpiredError 分支永遠不會被執行
- **檔案**: `backend/src/middleware/auth.ts:60-66`
- **問題**: `TokenExpiredError` 繼承自 `JsonWebTokenError`，先判斷父類就已匹配，導致過期 token 返回「無效 Token」而非「Token 已過期」，前端無法觸發 refresh 流程
- **影響**: 使用者 token 過期後被踢出登入，而非自動刷新
- **修復**: 把 `TokenExpiredError` 判斷移到 `JsonWebTokenError` 之前

### 7. [後端/資料庫] Refresh Token 和 OAuth State 存在記憶體
- **檔案**: `backend/src/services/authService.ts:10-26`
- **問題**: 服務重啟（每次 Zeabur 部署）會清空所有 refresh token，全員強制重新登入。多節點部署時 token 跨節點無效
- **影響**: 每次部署 = 全員登出，嚴重影響測試體驗
- **修復**: 遷移至 Redis 或新增 `refresh_tokens` 資料表

### 8. [前端] 四個頁面的 `/backlog` 路由連結已失效
- **檔案**: `DashboardPage.vue:121,154`、`MyTasksPage.vue:280`、`ReportPage.vue:378`
- **問題**: Router 中不存在 `/backlog`，正確路徑為 `/task-pool`。點擊「前往需求池」按鈕會導向 404
- **影響**: 核心導航功能壞掉
- **修復**: 全域替換 `to="/backlog"` 為 `to="/task-pool"`

### 9. [安全] 前端 Open Redirect 漏洞
- **檔案**: `packages/frontend/src/pages/LoginPage.vue:15-16`
- **問題**: `route.query.redirect` 未驗證即傳入 `router.push()`，攻擊者可構造惡意重導向 URL
- **修復**: 驗證 redirect 必須以 `/` 開頭且不含 `//`

### 10. [安全] Swagger API 文件在正式環境無保護公開
- **檔案**: `backend/src/index.ts:101`
- **問題**: `/api-docs` 在所有環境無需認證即可存取，等於完整 API 攻擊手冊
- **修復**: 生產環境停用或加上 Basic Auth 保護

### 11. [後端] authorizeTaskAccess 缺少 try-catch
- **檔案**: `backend/src/middleware/auth.ts:97-141`
- **問題**: async middleware 中執行 DB 查詢無 try-catch，Express 4 無法自動捕捉 async rejection，可能造成 request 掛死
- **修復**: 加 try-catch 或使用 `express-async-errors`

### 12. [後端] getPoolTasks 無分頁，全量載入所有任務
- **檔案**: `backend/src/services/taskService.ts:136-144`
- **問題**: `findMany` 不帶 `take/skip`，include 多個關聯，任務量大時記憶體爆炸
- **修復**: 加入分頁參數和 server-side 過濾

### 13. [後端] GitLab Issue webhook eventId 含 Date.now()，等冪性失效
- **檔案**: `backend/src/routes/gitlab/webhook.ts:298`
- **問題**: 每次 webhook 重送產生不同 eventId，導致重複建立 activity 記錄
- **修復**: 移除 `Date.now()`

### 14. [資料庫] batchConvertToTimeEntries 無 transaction 保護
- **檔案**: `backend/src/services/gitlab/activityService.ts:398-431`
- **問題**: 批次轉換逐筆呼叫無 transaction 的 `convertToTimeEntry()`，中途失敗造成部分成功的不一致狀態

### 15. [安全] Slack OAuth 登入缺少速率限制
- **檔案**: `backend/src/index.ts:87-89`
- **問題**: `authLimiter` 未覆蓋 `/api/auth/slack` 和 `/api/auth/refresh`

### 16. [安全] GitLab 加密 Salt 有可預測的硬編碼預設值
- **檔案**: `backend/src/utils/gitlab/encryption.ts:16`
- **問題**: 未設定時使用 `"progresshub-gitlab-default-salt"`，此值已公開在原始碼中

### 17. [DevOps] 後端 Dockerfile 使用 `npm install` 無鎖檔
- **檔案**: `backend/Dockerfile:13,37`
- **問題**: 無 `package-lock.json` 也不用 `npm ci`，每次建置可能安裝不同版本

### 18. [DevOps] 前端 production 建置啟用 source maps
- **檔案**: `packages/frontend/vite.config.ts:24`
- **問題**: `sourcemap: true` 無條件啟用，暴露完整原始碼結構

### 19. [前端] MainLayout Promise.all 預載全吞錯誤
- **檔案**: `packages/frontend/src/layouts/MainLayout.vue:33-50`
- **問題**: 8 個 store 資料並行載入，任何 API 失敗都被 catch 吞掉，使用者看到空白畫面無錯誤提示

## 審查總評

**整體架構品質：中上。** 專案的基礎建設做得扎實（Helmet、CORS、Rate Limiting、結構化日誌、Health Check、JWT 雙 token），程式碼分層清晰（Controller → Service → Prisma），前端的 Store/Composable 抽象設計合理。

**主要風險集中在三個領域：**

1. **安全性盲點**：Slack OAuth 的 isActive 繞過、Demo 按鈕暴露、Open Redirect — 這些在對內測試時就可能被發現並利用
2. **狀態持久化**：Refresh Token 存記憶體是最大的體驗殺手，每次部署全員登出
3. **Mock/真實資料切換不徹底**：TaskRelationCard 仍引用 mock、/backlog 路由未更新，說明從 mock 模式到 API 模式的遷移還有殘留

建議按三波行動清單逐步修復。第一波（5 項 P0）工作量不大，估計可在一天內完成。
